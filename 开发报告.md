# 网上书店微服务系统开发与运维报告

## 摘要

本课题设计并实现了一个基于Spring Cloud微服务架构的网上书店系统。系统采用前后端分离的开发模式，后端使用Spring Boot 3.2作为基础框架，集成Spring Cloud Gateway实现API网关、Consul实现服务注册与发现、RestTemplate结合LoadBalancer实现服务间负载均衡调用、jjwt库实现JWT无状态身份认证；前端使用Vue 3框架配合Element Plus组件库构建用户界面，并通过Axios拦截器实现JWT Token的自动携带与无感刷新。系统将业务拆分为认证服务、用户服务、商品服务、订单服务和网关服务五个独立微服务，每个服务拥有独立的数据库实现数据隔离，引入Redis缓存热点数据以提升读性能。在运维层面，系统采用Docker多阶段构建实现镜像瘦身，通过Docker Compose实现九容器的一键编排部署，编写了完整的Kubernetes部署清单支持生产环境的容器编排，并设计了Jenkins Pipeline实现持续集成与持续交付。本文将从需求分析、系统设计、模块实现、部署运维四个维度对项目进行完整阐述。

## 一、引言

### 1.1 项目背景

随着电子商务的快速发展，传统的单体架构已经难以满足业务的高并发、高可用和快速迭代需求。微服务架构通过将单一应用拆分为一组小型自治服务，每个服务围绕特定业务能力构建、独立部署、独立扩展，成为现代企业级应用的主流架构选择。

本课题选择「网上书店」作为业务场景，是因为它覆盖了电商系统的核心链路：商品浏览、搜索筛选、购物车、下单支付、订单管理、用户管理和权限控制，能够充分体现微服务架构的设计思想和技术挑战。

### 1.2 任务描述

依据课程要求，本系统需要实现以下核心目标：使用SpringBoot和SpringCloud开发一个微服务系统，包含基础的SpringBoot Web服务以及Consul服务注册发现、RestTemplate服务间访问、Gateway服务网关和JWT身份验证。在此基础上，完成DevOps流水线作业，通过GitLab管理系统源码，采用Jenkins完成系统的自动化集成与构建。系统需以容器化方式部署运行。

### 1.3 技术选型理由

在技术选型上，后端核心框架选用Spring Boot 3.2与Spring Cloud 2023.0.0。Spring Boot的自动配置和起步依赖极大简化了项目搭建，而Spring Cloud提供了一套完整的微服务治理方案。服务注册与发现选择Consul而非Eureka，是因为Consul具备更轻量的部署模式和内置的健康检查机制，其KV存储还可扩展为配置中心。API网关选择Spring Cloud Gateway而非Zuul，关键在于Gateway基于WebFlux响应式编程模型构建，在网关这种IO密集型场景下，其吞吐量和资源利用率显著优于基于Servlet同步模型的Zuul。服务间调用采用RestTemplate结合Spring Cloud LoadBalancer，这是Spring Cloud官方推荐的轻量级HTTP客户端方案，通过@LoadBalanced注解即可实现客户端负载均衡，无需额外的Ribbon依赖。

在数据访问层，ORM框架选择MyBatis-Plus而非JPA/Hibernate。MyBatis-Plus在保留原生SQL灵活性的同时，提供了Lambda表达式构建查询条件的能力，避免了字符串拼接导致的字段名硬编码问题，尤其适合分库分表场景下的复杂查询需求。

身份认证方案选择JWT而非传统的Session机制。在微服务架构中，各服务独立部署、无共享内存，Session机制需要引入Redis等外部存储实现共享，增加了系统复杂度和网络开销。JWT将用户信息编码在Token自身中，服务端无需存储状态，验证过程仅涉及签名校验和过期时间检查，天然适配微服务的无状态设计原则。

前端框架选择Vue 3而非React，考量在于Vue的渐进式设计使得学习者可以逐步引入其生态系统，配合Element Plus组件库能够快速构建出美观的管理后台界面。Vue 3的Composition API和响应式系统使得状态管理逻辑更加清晰，特别是Pinia状态管理库的设计理念与Vue 3高度契合。

在运维层面，容器化方案选择Docker而非直接部署，因为Docker解决了经典的环境不一致问题——开发环境、测试环境、生产环境之间的差异（JDK版本、系统库、配置文件路径等）常常导致「在我机器上能跑」的困境。Docker镜像将应用程序及其所有依赖打包在一起，确保在任何支持Docker的环境中行为一致。同时采用多阶段构建技术，在编译阶段使用包含完整JDK和Maven的镜像，在运行阶段仅使用精简的JRE镜像，将最终镜像体积从数百MB缩减至约200MB，既加快了镜像传输速度，也减少了运行时的攻击面。容器编排方面，本地开发和测试环境使用Docker Compose进行多服务的一键编排，生产环境准备了完整的Kubernetes部署清单，利用K8s的自愈、自动扩缩容和滚动更新能力保障服务的稳定运行。

## 二、需求分析

本系统面向两类用户角色：普通用户和管理员。普通用户的核心业务流程为：浏览图书、按分类或关键词搜索、查看图书详情、将图书加入购物车、填写收货信息提交订单、在个人订单页面查看订单状态并进行支付或取消操作。管理员的核心业务流程为：登录后进入独立的管理后台，通过仪表盘查看系统统计数据（图书总数、订单总数、用户总数、销售额），管理图书的上架与下架，维护分类信息，处理订单的状态流转（待支付→已支付→已发货→已完成），以及对用户进行禁用或启用操作。

权限模型采用基于角色的访问控制。在API层面，Gateway的JwtAuthFilter实现全局过滤：白名单路径（登录、注册、图书列表和详情等公开接口）直接放行；受保护路径需要请求头携带有效的Bearer Token。Token中编码了用户的角色信息，Gateway将用户ID、用户名和角色注入请求头（X-User-Id、X-Username、X-Role），下游微服务通过统一的Servlet过滤器提取这些信息并存入ThreadLocal上下文，业务代码无需感知认证细节即可获取当前用户。在前端层面，路由守卫根据localStorage中的角色字段控制页面可见性，非管理员用户访问/admin开头的路由将被重定向至首页。

## 三、系统总体设计

### 3.1 架构设计

系统的整体架构分为四层。最上层是Nginx反向代理，负责托管前端Vue应用的静态资源（HTML、JavaScript、CSS），并将以/api开头的请求代理转发至网关服务。第二层是Spring Cloud Gateway，作为整个后端微服务的唯一入口，承担路由转发、JWT认证、跨域处理三项职责。网关通过Consul的服务发现能力，以lb://前缀实现基于服务名的客户端负载均衡。第三层是业务微服务层，包含认证服务、用户服务、商品服务、订单服务四个独立进程，每个服务拥有独立的部署单元和端口。服务间存在横向调用关系：订单服务在创建订单时，通过RestTemplate调用商品服务获取图书的实时价格和库存信息，确保订单金额的准确性和库存的实时校验。第四层是数据层，MySQL按业务域拆分为三个独立数据库，Redis作为缓存层加速分类列表和图书详情等高频读操作。

整个系统的网络拓扑通过Docker自定义bridge网络bookstore-net互联，容器间通过服务名直接通信，无需暴露不必要的端口到宿主机。

### 3.2 数据库设计

数据库采用按业务维度分库的策略，这是微服务架构中数据治理的核心实践。共享数据库是微服务架构的反模式，因为它会导致服务间的隐式耦合——一个服务的数据库变更可能影响其他服务，违背了微服务的独立演进原则。

本系统将数据拆分为三个独立库。bookstore_user库服务于认证和用户两个微服务，其中认证服务负责写操作（用户注册、密码修改），用户服务负责读操作（用户信息查询），这是一种简单的读写分离策略——由于用户数据的写频率远低于读频率，两个服务共享同一数据库不会产生严重的资源竞争，而分库的意义在于未来可以将auth-service和user-service拆分为不同团队的独立服务，各自拥有独立数据源。bookstore_product库服务于商品微服务，存储图书表和分类表，读写比例约为10:1，因此在读路径上引入Redis缓存，采用Cache-Aside模式：查询时先访问缓存，缓存未命中则查询数据库并将结果回写缓存；写入时先更新数据库，然后删除缓存，由下一次读请求触发缓存的重新加载。bookstore_order库服务于订单微服务，存储订单主表和订单项明细表，订单数据涉及金额且变化频繁，不使用缓存，全部走数据库读写。

三库之间不存在外键约束——这是微服务分库的必然选择。跨库的数据一致性通过应用层保证：订单表中仅存储图书ID作为引用，不维护数据层面的外键关系。

### 3.3 JWT认证流程

JWT认证流程的设计是前后端分离架构中的关键环节，涉及Token的生成、传递和验证三个阶段的协议约定。

在Token生成阶段，用户通过POST请求将用户名和密码提交至/auth/login接口，认证服务验证凭据后，将一个包含userId、username、role三个自定义声明的JWT Token编码在HTTP响应体中返回。Token的有效期设置为24小时，签名算法采用HMAC-SHA512，密钥在配置文件中管理。选择将Token放在响应体而非响应头，是因为前端JavaScript无法访问Set-Cookie之外的自定义响应头，而JSON响应体可以被Axios拦截器统一处理。

在Token传递阶段，前端将Token存储在localStorage中，并通过Axios请求拦截器在每次请求的Authorization头中自动附加Bearer前缀的Token。选择请求头传递而非URL参数或POST参数，是因为URL参数会将Token暴露在服务器日志和浏览器历史中，存在安全隐患；HTTP请求头是RFC 6750 Bearer Token规范推荐的标准方式。

在Token验证阶段，Gateway的JwtAuthFilter作为全局过滤器拦截所有非白名单请求。过滤器首先检查白名单（登录、注册、图书列表等公开接口直接放行），然后从Authorization头提取Token，调用jjwt库进行签名验证和过期时间检查。验证通过后，过滤器将Token中的用户信息解码并注入到请求头X-User-Id、X-Username和X-Role中。下游微服务通过统一的ServiceUserContextFilter（Servlet Filter）从请求头中提取这些值并存入ThreadLocal，业务代码通过UserContext.getUserId()即可获取当前用户标识，做到认证与业务逻辑的彻底解耦。

## 四、模块详细设计与实现

### 4.1 公共模块

bookstore-common模块是所有微服务的共享基础，它不独立运行，而是作为JAR包被其他模块依赖。这种设计避免了代码重复——实体类、DTO、工具类和全局异常处理器只需编写一次。

模块内的核心组件包括：Result泛型响应类，封装了code（状态码）、message（提示信息）和data（泛型数据）三个字段，提供了一组静态工厂方法如success()、error()、unauthorized()等，使得Controller层的返回值始终遵循统一的JSON格式。JwtUtil工具类基于jjwt库封装了Token的生成、解析、验证和用户信息提取方法，采用静态方法设计，无需实例化。UserContext基于ThreadLocal实现请求级别的用户信息传递，利用ThreadLocal的线程隔离特性，确保并发请求之间的用户信息不会混淆，配合Filter在请求结束时的清理操作防止内存泄漏。GlobalExceptionHandler是一个@RestControllerAdvice切面，统一捕获并格式化处理BusinessException（业务异常）和Exception（系统异常），避免异常堆栈直接暴露给前端。

### 4.2 网关服务

bookstore-gateway是整个后端的唯一入口，基于Spring Cloud Gateway构建。Gateway基于WebFlux响应式编程模型，这意味着在处理并发请求时，它使用少量线程通过事件驱动的方式处理IO，而非为每个请求分配一个线程阻塞等待，因此能够以极低的资源消耗承载高并发流量。

路由配置采用服务发现模式：对于每个路由规则，通过lb://前缀指定目标服务在Consul中注册的名称，Gateway自动从Consul获取该服务的所有实例列表，并以轮询策略分发请求。路由规则覆盖了/auth/**、/user/**、/product/**和/order/**四个路径前缀。

JwtAuthFilter实现了GlobalFilter和Ordered接口。order值设为-100确保它在所有默认过滤器之前执行。白名单设计为：/api/auth/login、/api/auth/register、/api/product/books（含子路径如hot和详情）、/api/product/categories——这些是未登录用户即可访问的公共资源。对于受保护路径，过滤器解析Token后通过ServerHttpRequest.mutate()修改请求对象，将用户信息追加到请求头，然后传递给下游。这种方式避免了对请求体的修改，保持了对下游服务的透明性。

需要注意的一点是：网关中的UserContextFilter虽然将用户信息写入了当前线程的ThreadLocal，但Gateway的请求转发是异步非阻塞的，下游服务的响应到达时可能在不同的线程上，因此下游服务不能依赖网关设置的ThreadLocal。实际设计中，下游服务各自实现了自己的ServiceUserContextFilter来从请求头中读取用户信息。

### 4.3 认证服务

bookstore-auth-service处理用户的登录、注册和退出操作。登录时，服务通过MyBatis-Plus查询用户表，将用户输入的密码进行MD5加密后与数据库中的密文比对。验证通过后，调用JwtUtil生成Token，将Token与userId、username、role、nickname、avatar一同封装在Result响应中返回。

注册时，服务首先检查用户名是否已存在（通过MyBatis-Plus的LambdaQueryWrapper构建查询条件），然后对密码进行MD5加密，设置默认角色为USER，调用MyBatis-Plus的insert方法插入用户记录。退出操作不需要后端做特殊处理——JWT本身就是无状态的，不存在服务端销毁Token的概念；这个接口的存在主要是为了记录操作日志和清空前端的Token存储。

### 4.4 商品服务

bookstore-product-service是系统中业务逻辑最丰富的服务，负责图书和分类的增删改查。在查询层面，图书列表接口支持分页、关键词搜索（按书名、作者、ISBN模糊匹配）和分类筛选的多条件组合查询，使用MyBatis-Plus的LambdaQueryWrapper动态构建查询条件。每个图书对象在返回前会关联分类表查询分类名称并填充到categoryName字段中，避免了前端二次查询。

热门图书接口按创建时间倒序排列，取前N条记录，使用LIMIT语句直接在SQL层面限制结果集大小，而非Java层面截断，减少了数据传输量。

缓存策略在CategoryService中有完整的体现。分类列表的查询遵循Cache-Aside模式：先从Redis读取键category:all，命中则直接返回（避免了单次数据库查询和结果集到Java对象的转换开销）；未命中则查询数据库，将结果写入Redis并设置2小时的过期时间。在分类的增删改操作中，每次修改后立即删除缓存键，由下一次读取触发缓存的重新加载。这种策略保证了缓存与数据库的最终一致性，同时避免了对已缓存数据做复杂的逐条更新。

图书详情也采用了类似的缓存策略，缓存键为book:{id}，过期时间1小时，在图书更新或下架时删除对应缓存键。

### 4.5 订单服务

bookstore-order-service实现了下单、订单查询、支付、取消和状态管理功能。下单接口是这个服务最核心也是最复杂的业务逻辑，涉及服务间调用和事务处理。

创建订单时，前端提交的请求包含收货信息和订单项列表（每项包含图书ID和数量）。服务端首先从UserContext获取当前用户ID（由网关的JwtAuthFilter注入），然后生成格式为yyyyMMddHHmmss+6位UUID的订单号。接下来进入订单项处理循环：对每个订单项，通过RestTemplate调用商品服务的GET /api/product/books/{id}接口获取图书的实时价格和库存信息。这里选择以HTTP调用而非数据库联表查询的方式获取图书信息，体现了微服务数据隔离的原则——订单服务无权直接访问商品数据库。获取到实时价格后，校验库存是否充足，计算小计金额并累加到订单总额。

所有订单项处理完毕且验证通过后，在同一个Service方法上使用@Transactional注解开启数据库事务，依次插入订单主表和订单项明细表。事务保证了订单和订单项要么同时成功，要么同时回滚，避免了数据不一致。

订单的状态模型设计为有限状态机：0-待支付、1-已支付、2-已发货、3-已完成、4-已取消。状态流转遵循严格的路径约束——只有待支付状态（0）可以转为已支付（1）或已取消（4）；已支付（1）只能转为已发货（2）；已发货（2）只能转为已完成（3）；已取消（4）和已完成（3）是终态，不允许再变更。这些约束在Service层通过状态检查代码强制执行，防止了非法的状态跳转。

### 4.6 前端设计

前端项目的技术选型以Vue 3为核心，配套Vite构建工具、Element Plus UI组件库、Pinia状态管理和Vue Router路由管理。HTTP通信层使用Axios库，通过请求拦截器和响应拦截器实现了JWT Token的自动注入和异常处理。

在请求拦截器中，Axios从localStorage中获取Token，将其附加到每个请求的Authorization头中。这种方式使得所有通过request实例发出的API调用自动携带认证信息，业务代码无需重复处理。在响应拦截器中，当后端返回401状态码时，前端自动清除localStorage中的Token和用户信息，弹出提示框告知用户登录已过期，并重定向到登录页面。

路由设计分为两棵树：前台路由树（/为根节点）包含首页、图书列表、图书详情、购物车、我的订单、个人中心六个页面，共享同一个前台Layout布局；后台路由树（/admin为根节点）包含仪表盘、图书管理、分类管理、订单管理、用户管理五个页面，使用独立的管理后台AdminLayout布局。路由守卫在每次页面跳转时检查：未登录用户访问非公开页面时重定向至登录页；非管理员访问/admin路径时重定向至首页；已登录用户访问登录页时重定向至首页。

前端的权限区分在三个层面体现。第一层是路由守卫，阻止非管理员访问管理后台路由。第二层是导航栏渲染，Layout.vue中通过v-if指令判断用户角色，管理员可见橙色的管理后台入口按钮，普通用户则看不到。第三层是菜单项差异，管理员下拉菜单中额外包含管理后台选项。这种多层防护确保了权限控制既在体验层面清晰可见，又在技术层面不可绕过。

## 五、部署与运维

### 5.1 Docker多阶段构建

后端每个微服务的Docker镜像使用两阶段构建。第一阶段使用Maven 3.9 + JDK 17的Alpine镜像作为构建环境，利用Docker的构建缓存机制，先将POM文件复制到构建容器中并执行dependency:go-offline下载所有依赖——这层的缓存意味着只要POM文件不变，依赖下载就不会重复执行，大幅缩短了后续构建时间。然后将源码复制进去执行编译打包。第二阶段使用eclipse-temurin:17-jre-alpine作为运行时镜像，这是一个仅包含JRE的精简镜像，体积约180MB。从第一阶段将编译好的JAR包复制到运行时镜像中，并创建非root用户运行应用，遵循最小权限原则。

由于国内网络环境访问Maven中央仓库和Docker Hub存在困难，Dockerfile内置了Maven阿里云镜像配置，Docker Desktop配置了DaoCloud镜像加速器。这两个措施确保了在国内网络条件下依然能够正常完成镜像构建。

### 5.2 Docker Compose编排

Docker Compose将九个容器定义在一个YAML文件中，通过服务依赖关系和健康检查控制启动顺序。MySQL和Redis配置了健康检查——MySQL使用mysqladmin ping每10秒检测一次，Redis使用redis-cli ping每10秒检测一次——后端服务通过depends_on的condition: service_healthy参数确保在数据库完全就绪后才开始启动，避免了容器启动顺序竞争导致的连接失败。

所有容器通过自定义bridge网络bookstore-net连接，容器之间使用服务名作为主机名进行通信。这种网络隔离机制既保证了服务间的连通性，又防止了不必要的端口暴露——只有网关（8080）、前端（80）和Consul管理界面（8500）对外映射了端口。

### 5.3 Kubernetes部署方案

在Docker Compose的本地部署之外，项目还准备了完整的Kubernetes部署清单，用于生产环境的容器编排。部署清单包括以下资源定义：Namespace创建独立的bookstore命名空间实现资源隔离；ConfigMap存储非敏感的配置信息（数据库连接地址、Consul地址、Redis地址等），与容器镜像解耦，使得同一镜像在不同环境中的行为由ConfigMap决定；Secret存储数据库密码和JWT密钥等敏感信息，内容以Base64编码存储，Pod中的容器通过环境变量注入读取；MySQL和Redis使用Deployment+PersistentVolumeClaim的组合保障数据持久性，PV独立于Pod的生命周期，Pod重建后数据不丢失；Consul使用StatefulSet部署以确保稳定的网络标识和有序的扩缩容；各后端微服务使用Deployment管理多副本，通过Service的ClusterIP实现内部负载均衡；Ingress配置域名路由规则，将外部流量分发到对应的Service。

### 5.4 CI/CD流水线

Jenkinsfile定义了完整的持续集成与持续交付流水线。流水线分为六个阶段：Checkout阶段从GitLab拉取最新源码；Build Backend阶段使用Maven编译所有后端模块并运行单元测试；Build Frontend阶段使用npm安装依赖并执行Vite构建；Docker Build阶段为每个微服务构建Docker镜像；Docker Push阶段将镜像推送至私有Docker Registry；Deploy阶段根据目标环境（开发/生产）执行对应的Docker Compose或kubectl部署命令。每个阶段的失败都会触发邮件通知，确保团队能够及时响应构建问题。

### 5.5 运维命令参考

日常运维中常用的操作包括：查看全部容器状态使用docker compose ps，查看某个服务的日志使用docker compose logs -f {service-name}，重启单个服务使用docker compose restart {service-name}，完全重建所有服务使用docker compose down && docker compose up -d，备份数据库使用docker exec bookstore-mysql mysqldump -uroot -p123456 --all-databases > backup.sql，恢复数据库使用docker exec -i bookstore-mysql mysql -uroot -p123456 < backup.sql。

## 六、项目总结

本项目严格依据课程要求，完成了一个涵盖Spring Boot Web服务开发、Consul服务注册与发现、RestTemplate服务间调用、Gateway网关路由、JWT身份认证五大核心技术点的微服务系统。在开发过程中，深入理解了微服务架构的核心设计原则——服务拆分与数据隔离、无状态认证、集中化网关、声明式服务间调用——并将这些原则落实到代码层面。

在运维层面，通过Docker多阶段构建、Docker Compose服务编排、Kubernetes部署清单和Jenkins流水线，构建了一套从代码提交到生产部署的完整DevOps链路。这套链路体现了现代软件工程的核心价值观：自动化、可重复、可追溯。

系统当前已实现了网上书店的核心业务闭环，并严格区分了管理员和普通用户的功能边界。后续可以在分布式事务、链路追踪、消息队列和配置中心等方向继续深化，进一步提升系统的健壮性和可观测性。
