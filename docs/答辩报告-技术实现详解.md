## 1. Spring Boot Web 服务开发

### 1.1 它是什么

Spring Boot 是 Java 生态中最主流的企业级开发框架。它的核心价值在于**自动配置**——你只需引入一个 starter 依赖，框架就会自动帮你配置好数据库连接、Web 服务器、序列化等所有基础设施。你只需要写业务代码。

### 1.2 本项目怎么实现的

系统拆分为 **5 个独立的 Spring Boot 应用**，每个都是一个独立的进程，拥有自己的端口、配置和数据库：

| 服务 | 端口 | 职责 | 数据库 |
|------|------|------|--------|
| bookstore-gateway | 8080 | API 网关 · 路由 · JWT 认证 | 无 |
| bookstore-auth-service | 8081 | 登录 · 注册 · Token 签发 | bookstore_user |
| bookstore-user-service | 8082 | 用户信息 · 地址管理 | bookstore_user |
| bookstore-product-service | 8083 | 图书 CRUD · 分类 · 缓存 | bookstore_product |
| bookstore-order-service | 8084 | 下单 · 支付 · 订单状态 | bookstore_order |

每个服务都是一个标准的 Spring Boot 项目结构：

```
bookstore-xxx-service/
├── src/main/java/com/bookstore/xxx/
│   ├── XxxApplication.java      ← 启动类
│   ├── controller/               ← REST 接口
│   ├── service/                  ← 业务逻辑
│   ├── mapper/                   ← 数据库访问 (MyBatis-Plus)
│   └── entity/                   ← 数据实体
└── src/main/resources/
    └── application.yml           ← 配置文件
```

### 1.3 关键代码

**启动类示例（OrderServiceApplication.java）：**

**REST 接口示例（AuthController.java 中的登录接口）：**

## 2. Consul 服务注册与发现

### 2.1 它是什么

Consul 是一个**服务注册中心**。在微服务架构中，服务之间需要互相调用，但每个服务可能有很多个实例，IP 和端口也可能变化。Consul 的作用就像一个**电话本**——每个服务启动时把自己的地址注册到 Consul，调用方通过服务名去 Consul 查到实际的地址。

### 2.2 本项目怎么实现的

**第一步：每个服务配置 Consul 地址**

每个微服务的 `application.yml` 中都有相同的配置块：

```yaml
spring:
  application:
    name: bookstore-auth-service    # 服务名，注册到 Consul 的唯一标识
  cloud:
    consul:
      host: localhost               # Consul 地址
      port: 8500                    # Consul 端口
      discovery:
        service-name: ${spring.application.name}
        health-check-path: /actuator/health    # 健康检查路径
        health-check-interval: 10s             # 每10秒检查一次
```

**第二步：启动类加注解**

```java
@SpringBootApplication
@EnableDiscoveryClient    // ← 这个注解让服务启动时自动注册到 Consul
public class AuthServiceApplication { ... }
```

**第三步：Gateway 通过服务名发现服务**

Gateway 的路由配置中用 `lb://服务名` 格式，自动从 Consul 获取实例列表：

```yaml
# Gateway 的 application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: product-service
          uri: lb://bookstore-product-service    # ← 通过 Consul 发现
          predicates:
            - Path=/api/product/**
```

**文件位置：**
- `backend/bookstore-auth-service/src/main/resources/application.yml`
- `backend/bookstore-gateway/src/main/resources/application.yml`
- `backend/bookstore-order-service/src/main/resources/application.yml` (同上模式)

### 2.3 怎么验证

1. 打开浏览器访问 **http://localhost:8500** → Consul 管理界面
2. 点击 **Services** → 看到 5 个绿色对勾的服务
3. 每个服务的健康检查状态都是 passing
4. 这说明所有服务都在启动时自动注册到 Consul，并且持续汇报健康状态

## 3. RestTemplate 服务间调用

### 3.1 它是什么

微服务之间不能像单体应用那样直接调用方法（因为不在同一个进程里），需要通过 **HTTP 请求**来互相通信。RestTemplate 是 Spring 提供的 HTTP 客户端，用来发起 HTTP 请求。加上 `@LoadBalanced` 注解后，它就能通过服务名（而不是 IP 地址）来调用——它会自动去 Consul 查到对方的实际地址。

### 3.2 本项目怎么实现的

**场景：** 用户下单时，订单服务需要知道每本图书的**实时价格**和**当前库存**。但图书数据在商品服务自己的数据库里，订单服务不能直接访问。

**第一步：配置 RestTemplate Bean（加上负载均衡能力）**

```java
// OrderServiceApplication.java
@Bean
@LoadBalanced    // ← 关键：让 RestTemplate 能解析服务名
public RestTemplate restTemplate() {
    return new RestTemplate();
}
```

**第二步：通过服务名（非 IP）发起 HTTP 调用**

```java
// OrderService.java — 创建订单时的服务间调用
private Book getBookInfo(Long bookId) {
    // 注意这里：URL 用的是服务名，不是 IP 地址！
    String url = "http://bookstore-product-service/api/product/books/" + bookId;

    ResponseEntity<Result> response = restTemplate.exchange(
        url, HttpMethod.GET, null,
        new ParameterizedTypeReference<Result>() {}
    );

    // 从响应中拿到图书的实时价格和库存
    Result result = response.getBody();
    Map<String, Object> bookData = (Map<String, Object>) result.getData();
    // ...用 bookData 计算订单金额、校验库存
}
```

**调用链路图：**

```
订单服务                                    Consul                   商品服务
  │                                          │                        │
  │ 1. 需要查询图书 #5                       │                        │
  │ 2. 问 Consul："product-service 在哪？"    │                        │
  │────────────────────────────────────────→│                        │
  │ 3. Consul 返回：":8083 机器A"            │                        │
  │←────────────────────────────────────────│                        │
  │ 4. GET http://192.168.1.5:8083/api/product/books/5               │
  │─────────────────────────────────────────────────────────────────→│
  │ 5. 返回 { price: 59.00, stock: 12 }                              │
  │←─────────────────────────────────────────────────────────────────│
```

### 3.3 关键对比

| 方式 | 代码 | 问题 |
|------|------|------|
| ❌ 硬编码 IP | `http://192.168.1.5:8083/...` | IP 变了就崩 |
| ❌ 直连数据库 | `jdbc:mysql://.../bookstore_product` | 破环数据隔离 |
| ✅ **RestTemplate + Consul** | `http://bookstore-product-service/...` | 自动发现，数据隔离 |

**文件位置：**
- Bean 定义：`backend/bookstore-order-service/src/main/java/com/bookstore/order/OrderServiceApplication.java`
- 调用代码：`backend/bookstore-order-service/src/main/java/com/bookstore/order/service/OrderService.java` 中 `getBookInfo()` 方法

### 3.4 怎么验证

1. 登录后添加商品到购物车 → 下单
2. F12 → Network → 只有 `/api/order` 一个请求（前端只调了订单接口）
3. 但订单金额和库存校验都正确 → 说明订单服务内部通过 RestTemplate 成功调用了商品服务

## 4. Gateway 服务网关

### 4.1 它是什么

Gateway 是整个后端系统的**唯一入口**。所有前端请求只发给 Gateway（端口 8080），Gateway 再根据 URL 路径把请求转发给对应的微服务。

就像公司的**前台**——访客只需要知道前台的电话，前台会根据访客要找谁转接到对应部门。

### 4.2 本项目怎么实现的

Gateway 做了三件事：**路由转发**、**JWT 认证**、**跨域处理**。

**第一件事：路由转发**

```yaml
# Gateway 的 application.yml
server:
  port: 8080

spring:
  cloud:
    gateway:
      routes:
        - id: auth-service
          uri: lb://bookstore-auth-service       # lb:// = 从 Consul 发现
          predicates:
            - Path=/api/auth/**                   # 匹配 /api/auth/ 开头的请求

        - id: user-service
          uri: lb://bookstore-user-service
          predicates:
            - Path=/api/user/**

        - id: product-service
          uri: lb://bookstore-product-service
          predicates:
            - Path=/api/product/**

        - id: order-service
          uri: lb://bookstore-order-service
          predicates:
            - Path=/api/order/**
```

**效果：** 前端只需知道 `http://localhost:8080`，Gateway 自动分发。

**第二件事：JWT 认证（全局过滤器）**

Gateway 在转发之前拦截每个请求，检查是否带了合法的 JWT Token：

```java
// JwtAuthFilter.java
@Component
public class JwtAuthFilter implements GlobalFilter, Ordered {

    // 白名单：这些路径不需要登录就能访问
    private static final List<String> WHITELIST = List.of(
        "/api/auth/login",
        "/api/auth/register",
        "/api/product/books",    // 图书列表公开
        "/api/product/categories" // 分类列表公开
    );

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getURI().getPath();

        // 1. 白名单路径直接放行
        if (isWhitelist(path)) {
            return chain.filter(exchange);
        }

        // 2. 从请求头取 Token
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (token == null || !token.startsWith("Bearer ")) {
            return unauthorizedResponse(exchange, "未提供认证令牌");
        }
        token = token.substring(7);  // 去掉 "Bearer " 前缀

        // 3. 验证 Token（签名 + 过期时间）
        if (!JwtUtil.validateToken(token)) {
            return unauthorizedResponse(exchange, "令牌无效或已过期");
        }

        // 4. 解析用户信息，注入到请求头传给下游服务
        Claims claims = JwtUtil.parseToken(token);
        exchange = exchange.mutate()
            .request(r -> r.header("X-User-Id", claims.get("userId").toString()))
            .request(r -> r.header("X-Username", claims.get("username").toString()))
            .request(r -> r.header("X-Role", claims.get("role").toString()))
            .build();

        // 5. 放行
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() {
        return -100;  // 最高优先级，在其他过滤器之前执行
    }
}
```

**第三件事：跨域处理（CORS）**

```java
// GatewayApplication.java
@Bean
public WebFilter corsFilter() {
    return (exchange, chain) -> {
        ServerHttpRequest request = exchange.getRequest();
        ServerHttpResponse response = exchange.getResponse();
        response.getHeaders().add("Access-Control-Allow-Origin", "*");
        response.getHeaders().add("Access-Control-Allow-Methods", "GET, POST, PUT, DELETE, OPTIONS");
        response.getHeaders().add("Access-Control-Allow-Headers", "Authorization, Content-Type");
        if (request.getMethod() == HttpMethod.OPTIONS) {
            response.setStatusCode(HttpStatus.OK);
            return Mono.empty();
        }
        return chain.filter(exchange);
    };
}
```

**文件位置：**
- 路由配置：`backend/bookstore-gateway/src/main/resources/application.yml`
- JWT 过滤器：`backend/bookstore-gateway/src/main/java/com/bookstore/gateway/filter/JwtAuthFilter.java`
- CORS 配置：`backend/bookstore-gateway/src/main/java/com/bookstore/gateway/GatewayApplication.java`

## 5. JWT 身份验证

### 5.1 它是什么

JWT（JSON Web Token）是一种**无状态**的身份认证方案。传统方案用 Session：登录后服务端存一个 Session ID，每次请求都要查 Session。但在微服务架构中，服务之间不共享内存，Session 方案需要引入 Redis 等外部存储，增加复杂度。

JWT 的方案是：登录后服务端签发一个**加密的令牌**，用户信息直接编码在令牌里。后续请求携带这个令牌，服务端只需验证签名和过期时间，不需要查数据库。

### 5.2 本项目怎么实现的

**完整流程：**

```
[前端]                      [Gateway]                   [AuthService]
  │                            │                            │
  │ 1. POST /api/auth/login    │                            │
  │ {username, password}       │                            │
  │───────────────────────────→│ 路由到 auth-service        │
  │                            │───────────────────────────→│
  │                            │                            │ 2. 查数据库校验密码
  │                            │                            │ 3. 调用 JwtUtil 签发 Token
  │                            │ 4. 返回 Token              │
  │                            │←───────────────────────────│
  │ 5. 收到 Token              │                            │
  │←───────────────────────────│                            │
  │                            │                            │
  │ 6. 存 localStorage         │                            │
  │ 7. 后续请求带 Token        │                            │
  │ Authorization: Bearer xxx  │                            │
  │───────────────────────────→│ 8. JwtAuthFilter 验证      │
  │                            │ 9. 解析出 userId/role      │
  │                            │ 10. 注入 X-User-* 请求头    │
  │                            │───────────────────────────→│ 下游服务直接用
```

**第一步：登录时签发 JWT**

```java
// AuthService.java — 登录方法
public Map<String, Object> login(LoginDTO loginDTO) {
    // 1. 查用户（MyBatis-Plus Lambda 查询）
    User user = userMapper.selectOne(
        new LambdaQueryWrapper<User>()
            .eq(User::getUsername, loginDTO.getUsername())
    );
    if (user == null) {
        throw new BusinessException("用户名或密码错误");
    }

    // 2. 校验密码（MD5 加密后比对）
    String encryptedPwd = DigestUtils.md5DigestAsHex(
        loginDTO.getPassword().getBytes()
    );
    if (!encryptedPwd.equals(user.getPassword())) {
        throw new BusinessException("用户名或密码错误");
    }

    // 3. 签发 JWT Token
    String token = JwtUtil.generateToken(
        user.getId(),        // 用户ID
        user.getUsername(),  // 用户名
        user.getRole()       // 角色（USER / ADMIN）
    );

    // 4. 返回 Token + 用户信息
    Map<String, Object> result = new HashMap<>();
    result.put("token", token);
    result.put("userId", user.getId());
    result.put("username", user.getUsername());
    result.put("role", user.getRole());
    return result;
}
```

**第二步：JWT 工具类（生成 + 验证）**

```java
// JwtUtil.java
public class JwtUtil {
    private static final String SECRET = "bookstore-secret-key-2024-microservice-architecture";
    private static final long EXPIRATION = 24 * 60 * 60 * 1000;  // 24小时
    private static final SecretKey SECRET_KEY =
        Keys.hmacShaKeyFor(SECRET.getBytes(StandardCharsets.UTF_8));

    // 生成 Token
    public static String generateToken(Long userId, String username, String role) {
        return Jwts.builder()
            .claim("userId", userId)        // 自定义字段：用户ID
            .claim("username", username)    // 自定义字段：用户名
            .claim("role", role)            // 自定义字段：角色
            .setSubject(username)
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + EXPIRATION))
            .signWith(SECRET_KEY)           // HMAC-SHA512 签名
            .compact();
    }

    // 验证 Token（返回解析后的 Claims，失败抛异常）
    public static Claims parseToken(String token) {
        return Jwts.parser()
            .verifyWith(SECRET_KEY)
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }

    // 简单校验：能解析就是有效
    public static boolean validateToken(String token) {
        try {
            parseToken(token);
            return true;
        } catch (Exception e) {
            return false;
        }
    }
}
```

**第三步：前端自动携带 Token（Axios 拦截器）**

```javascript
// frontend/src/utils/request.js
const request = axios.create({
  baseURL: '/api',
  timeout: 10000,
})

// 请求拦截器：每个请求自动加上 Token
request.interceptors.request.use(config => {
  const token = localStorage.getItem('token')
  if (token) {
    config.headers.Authorization = `Bearer ${token}`   // ← 自动注入
  }
  return config
})

// 响应拦截器：Token 过期自动跳转登录
request.interceptors.response.use(
  response => response,
  error => {
    if (error.response?.status === 401) {
      localStorage.removeItem('token')
      localStorage.removeItem('user')
      window.location.href = '/login'
    }
    return Promise.reject(error)
  }
)
```

### 5.3 为什么要选 JWT 而不是 Session

| | Session | JWT |
|------|---------|-----|
| 服务端存储 | 需要存（Redis/内存） | 不需要（无状态） |
| 微服务共享 | 需要额外中间件 | 天然兼容 |
| 水平扩展 | 需要共享存储 | 任意扩展 |
| 每次请求开销 | 查 Redis | 仅验签（纯计算） |

**文件位置：**
- JWT 工具类：`backend/bookstore-common/src/main/java/com/bookstore/common/utils/JwtUtil.java`
- 登录逻辑：`backend/bookstore-auth-service/src/main/java/com/bookstore/auth/service/AuthService.java`
- Gateway 过滤器：`backend/bookstore-gateway/src/main/java/com/bookstore/gateway/filter/JwtAuthFilter.java`
- 前端拦截器：`frontend/src/utils/request.js`

2. F12 → Network → 点任意请求 → Request Headers → 看到 `Authorization: Bearer eyJ...`

## 7. Jenkins 自动化构建

### 7.1 它是什么

Jenkins 是一个 **CI/CD（持续集成/持续交付）** 工具。它的作用是把代码从提交到部署的全过程自动化。没有 Jenkins 时，每次代码变更需要手动执行：编译 → 测试 → 打包 → 构建镜像 → 部署。有了 Jenkins Pipeline，这些步骤在代码推送后全自动执行。

### 7.2 本项目怎么实现的

项目根目录的 `Jenkinsfile` 定义了完整的 **6 阶段流水线**：

```groovy
pipeline {
    agent any

    stages {
        // 阶段1：拉取代码
        stage('Checkout') {
            steps {
                checkout scm     // 从 GitLab 拉取最新代码
            }
        }

        // 阶段2：编译后端（Maven 打包所有微服务）
        stage('Build Backend') {
            steps {
                sh 'mvn clean package -DskipTests'
            }
        }

        // 阶段3：编译前端（npm + Vite）
        stage('Build Frontend') {
            steps {
                dir('frontend') {
                    sh 'npm install'
                    sh 'npm run build'
                }
            }
        }

        // 阶段4：构建 Docker 镜像（多阶段构建）
        stage('Docker Build') {
            steps {
                sh 'docker compose build'
            }
        }

        // 阶段5：推送镜像到私有仓库
        stage('Docker Push') {
            steps {
                sh 'docker compose push'
            }
        }

        // 阶段6：部署到目标环境
        stage('Deploy') {
            steps {
                sh 'docker compose up -d'  // 开发环境用 Compose
                // 生产环境用 kubectl apply -f k8s/
            }
        }
    }

    post {
        failure {
            // 构建失败发邮件通知
            emailext body: '构建失败，请检查', subject: 'Pipeline Failed',
                     to: 'dev-team@example.com'
        }
    }
}
```

**文件位置：** `Jenkinsfile`（项目根目录）

### 7.3 流水线可视化

```
代码提交 → [Checkout] → [Build Backend] → [Build Frontend] → [Docker Build] → [Push] → [Deploy]
             拉代码        Maven 编译        npm + Vite       镜像构建        推送仓库     启动容器
```

## 8. Docker 容器化部署

### 8.1 它是什么

Docker 把应用程序及其所有依赖（JDK、系统库、配置文件等）打包成一个**镜像**，然后在任何安装了 Docker 的机器上运行。这解决了经典的「在我机器上能跑」问题——开发环境、测试环境、生产环境完全一致。

### 8.2 本项目怎么实现的

**第一步：多阶段构建 Dockerfile（一个文件构建所有后端服务）**

```dockerfile
# ========== 阶段1：构建阶段 ==========
FROM maven:3.9-eclipse-temurin-17-alpine AS builder

# 配置阿里云 Maven 镜像加速
COPY docker/maven-settings.xml /root/.m2/settings.xml

WORKDIR /build

# 先复制 POM 文件（利用 Docker 缓存：只要 POM 不变，就不重新下载依赖）
COPY pom.xml .
COPY bookstore-common/pom.xml bookstore-common/
COPY bookstore-gateway/pom.xml bookstore-gateway/
COPY bookstore-auth-service/pom.xml bookstore-auth-service/
COPY bookstore-user-service/pom.xml bookstore-user-service/
COPY bookstore-product-service/pom.xml bookstore-product-service/
COPY bookstore-order-service/pom.xml bookstore-order-service/

# 下载依赖（这层会被缓存）
RUN mvn dependency:go-offline -pl ${SERVICE_NAME} -am

# 复制源码并编译
COPY . .
RUN mvn clean package -pl ${SERVICE_NAME} -am -DskipTests

# ========== 阶段2：运行阶段（仅 JRE，无 JDK/Maven） ==========
FROM eclipse-temurin:17-jre-alpine

# 创建非 root 用户（安全最小权限原则）
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# 只从构建阶段复制最终的 JAR 包
ARG SERVICE_NAME
COPY --from=builder /build/${SERVICE_NAME}/target/${SERVICE_NAME}-1.0.0.jar app.jar

USER appuser

ENTRYPOINT ["java", "-jar", "/app/app.jar"]
```

**关键设计点：**

| 设计 | 作用 |
|------|------|
| `ARG SERVICE_NAME` | 一个 Dockerfile 构建所有服务（传入不同服务名） |
| 先 COPY POM → 再 COPY 源码 | 利用 Docker 缓存，依赖不变时秒级构建 |
| `maven:3.9` → `jre-alpine` | 构建镜像 ~500MB，最终镜像仅 ~200MB |
| 非 root 用户运行 | 防止容器逃逸攻击 |

**第二步：Docker Compose 一键编排 9 个容器**

```yaml
# docker-compose.yml（简化版）
services:
  consul:
    image: consul:1.15
    ports: ["8500:8500"]

  mysql:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: 123456
    volumes:
      - ../sql/init.sql:/docker-entrypoint-initdb.d/init.sql  # 自动建库建表
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]

  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]

  auth-service:
    build:                          # 用上面的 Dockerfile 构建
      context: ../backend
      dockerfile: docker/Dockerfile
      args:
        SERVICE_NAME: bookstore-auth-service
    depends_on:                     # 等 MySQL + Consul 就绪才启动
      mysql: { condition: service_healthy }
      consul: { condition: service_started }

  user-service:     # ... 同上
  product-service:  # ... 同上
  order-service:    # ... 同上
  gateway-service:  # ... 同上

  frontend:
    build:
      context: ../frontend
      dockerfile: docker/Dockerfile
    ports: ["80:80"]
```

**启动命令：**

```bash
docker compose up -d       # 一键启动所有容器
docker compose ps           # 查看9个容器状态
docker compose down         # 停止所有容器
docker compose logs -f gateway-service   # 查看网关日志
```
